LLM · RAG · Skill · Agent

四個常被混用的詞，其實是一層疊一層的積木。

它們不是彼此的競爭者，而是能力的堆疊。LLM 是會思考的大腦；RAG 給它可查的資料；Skill 給它按需載入的做事流程；Agent 給它手腳，加上一個「自己會重複」的工作循環。分清楚，才知道什麼問題該加哪一層。

核心引擎

LLM 單次問答

輸入 prompt，輸出一段文字。沒有記憶、沒有動作、問完就結束。

加上循環

Agent 自主循環

思考 → 用工具 → 看結果 → 再思考，反覆執行直到目標達成。這一圈就是 Agent 的本質。

逐一拆解

四塊積木，各自解決什麼？

用「把 LLM 培養成一個能幹的知識工作者」來類比，每塊積木補上一項它天生缺的能力。

The Brain

LLM

Large Language Model

會推理、會語言的大腦本體

是什麼
吃海量文字訓練出的模型，本質是預測下一個字，因而能對話、寫作、推理。
強
通用語言與推理能力，開箱即用。
天生缺
只知道訓練時的資料、會過期、會編造；沒有記憶、不能動手做事。
例子
Claude、GPT、Gemini。

The Open Book

RAG

Retrieval-Augmented Generation

回答前先去查你的資料，再作答

是什麼
問題進來時，先從你的知識庫 / 文件 / 資料庫檢索相關片段，塞進 prompt 再讓 LLM 作答。
解決
模型不知道你家內部資料、資料過期、幻覺。等於從閉卷變開卷考試。
不改
不改模型本身，只是餵它更好的 context。
例子
問公司內規、查最新文件、客服知識庫。

The Playbook

Skill

可按需載入的能力包

教它「這類任務照這套 SOP 做」

是什麼
把某個領域的指令、流程、範本、腳本打包，模型判斷相關時才載入，用完釋放。
解決
把「怎麼做這件事」的專業知識與規範，穩定注入模型行為。
關鍵
progressive disclosure：平時只看一行描述，需要時才展開全文，省 context。
例子
Claude Code 的 skill：翻譯、code review、投影片。

The Worker

AI Agent

會自主行動的執行體

給大腦裝上手腳與工作循環

是什麼
LLM ＋ 工具（tools）＋ 一個迴圈：自己決定下一步、呼叫工具、看結果、修正，直到完成目標。
解決
不再只是回一段話，而是能真的去做多步驟任務。
常含
常會用到 RAG 與 Skill 當作它的工具與能力。
例子
Claude Code 自己讀檔、跑指令、改 code、驗證。

組合關係

它們怎麼疊起來

由內而外，每一層都建立在前一層之上。越上層越自主，也越複雜。

L0

LLM 基底

一切的核心。單次輸入輸出，靠內建知識回答。

提供：語言 + 推理

L1

LLM + RAG 加外部知識

回答前先檢索你的資料。大腦沒變，但變成開卷、答得準、能引用來源。

＋ 即時 / 私有知識

L1

LLM + Skill 加做事流程

需要時載入特定領域的 SOP 與範本，讓輸出穩定符合規範。與 RAG 同層、正交，可並存。

＋ 專業流程 / 一致性

L2

AI Agent 加自主循環

在上面全部之上，加一個「思考→行動→觀察」的迴圈與一組工具。RAG 是它的一個查資料工具，Skill 是它的一項能力，LLM 是它的大腦。

＋ 工具 + 迴圈 + 自主

一句話記住： Agent 是「容器」，LLM / RAG / Skill 都可能被裝在裡面。所以它們不是四選一。Agent = LLM + (RAG?) + (Skill?) + tools + loop。

同一個例子 · 貫穿四層

做一個「員工報帳助理」

用同一個場景，看每加一層改變了什麼。這是分辨差異最快的方法。

只有 LLM

憑記憶答，會唬爛 L0

你問「餐費上限多少？」它憑訓練資料回答，很可能編一個你公司根本沒有的數字。它不知道你們家的規章。

+ RAG

先查你的文件再答 +知識

接上公司報帳規章知識庫。問句進來時先檢索「餐費上限」條文，塞進 prompt 才作答，並能引用真實條文。治「不知道你的資料 + 幻覺」。但它還不會自己做事。

+ Skill

每次都照同一套 SOP +流程

打包一個「報帳審核 SOP」skill：怎麼判科目、要哪些必填欄位、輸出成什麼格式。每次審單都照同一套流程、同一種格式。治「流程不穩、缺規範」。RAG 與 Skill 可以並存，一個給知識、一個給做法。

= Agent

自己把整件事做完 +迴圈

給它工具（收據 OCR、查 ERP、寫入報帳系統）＋一個迴圈。出錯自己重試、模糊自己判斷或問你。此時 RAG 是它的查資料工具、Skill 是它的能力、LLM 是它的大腦，四者合體，不是四選一。

讀收據(OCR) → 查規則(RAG) → 按 SOP 分類(Skill) → 填進報帳系統(tool) → 超標就退回 → 回報

怎麼選

我的問題該加哪一層？

從「你缺的是什麼」倒推，通常一次只加一層，別一開始就上 Agent。

我只需要通用的理解與生成

純 LLM 就夠

→ 直接呼叫模型

寫作、翻譯、摘要、腦力激盪這類「一問一答」，不需要外部資料也不用動手，加更多層只是增加成本與複雜度。

模型不知道我的資料 / 答案會過期

加 RAG

→ 接上知識庫檢索

客服問內規、查最新文件、要能引用來源。想降低幻覺、又不想重訓模型時的首選。

同一類任務要每次照規範穩定做

加 Skill

→ 打包流程與範本

把「這種任務的正確做法」固化成能力包，需要時載入，讓輸出格式與品質一致、可重用。

任務有很多步、要自己完成到底

做成 Agent

→ 給工具 + 迴圈

要它自己查、自己做、自己驗、出錯自己修。這時才需要迴圈與自主，並把 RAG / Skill 當成它手上的工具。

別再搞混

兩組最常被混淆的對照

Skill 附近有兩個容易撞在一起的概念，講清楚界線。

Skill vs Tools / MCP

Skill教它「這類任務怎麼做」—— 指令、流程、範本。
Tools / MCP給它「動作的接口」—— 讀寫系統、呼叫 API。MCP 是接外部工具的標準協定。
關係一個是知識、一個是手；Agent 兩者都會用到。

Skill vs Fine-tuning

Skill不改模型，隨插即用、隨時增修，需要時才載入。
Fine-tuning改模型權重，成本高、要資料與訓練，改一次是永久內建。
先用誰多數情況先用 Skill / RAG；非到「行為要內建」才 fine-tune。
